# Git Lesson

## リモートリポジトリとローカルリポジトリとはそれぞれ何でしょうか？
  ### リモートリポジトリ（みんなで共有する場所）
  - リモートリポジトリは、ネットワーク上のサーバーに置かれたリポジトリです。GitHubなどが含まれます。
    - 役割：チームメンバー間での情報の共有や、プロジェクトのバックアップを行うための拠点です。
    - 特徴：
      - インターネットを通じて、離れた場所にいるメンバーと最新のコードを同期できます。
      - 他のメンバーが作成した修正を取り込んだり、自分の作業を公開したりするために使います。
      - プロジェクトの「正解（最新版）」を管理する場所として機能します。

  ### ローカルリポジトリ（自分の手元の場所）
  - ローカルリポジトリは、自身のパソコン内に作成されるリポジトリです。
    - 役割：プロジェクトの作業場所です。
    - 特徴：
      - ネットワークに接続していなくても、ファイルの編集や履歴の保存（コミット）が可能です。
      - 自分一人で変更を試したり、検証したりするのに適しています。
      - あくまで自分のパソコン内にあるため、他の人は直接見ることができません。

## プッシュとマージの違いは何でしょうか？
  ### プッシュ(Push)：「場所」を移動させる操作（ローカル → リモート）
  - プッシュは、自分のローカルリポジトリで行った変更履歴を、リモートリポジトリ（GitHubなど）へ送信して同期する操作です。
    - 目的：自分の作業をサーバーに保存したり、チームメンバーに共有したりすること。
    - イメージ：「自分のPCにある作業結果を、みんなが見られるネット上の保管庫にアップロードする」

  ### マージ（Merge）：「内容」を統合する操作（ブランチA + ブランチB → 合体）
  - あるブランチ（枝分かれした作業内容）の変更分を、別のブランチに取り込んで一本化する操作です。
    - 目的：別々に行っていた作業（例：機能追加の作業と、バグ修正の作業）を一つの場所にまとめること。
    - 特徴：「並行して作業していた2つのプログラムのコードを、1つのファイルに統合する」

## コミットとプッシュの違い
  ### コミット（Commit）：「自分のPCの中」に保存（記録）する操作
  - コミットは、現在のファイルの変更内容を、自分のPC（ローカルリポジトリ）に「変更履歴」として保存する操作です。
    - 目的：作業の区切りを記録する。過去のどの時点にでも戻れるようにする。
    - 特徴：　
      - 繰り返しても、他の人には全く影響しません。
      - インターネットに繋がっていなくても実行できます。
      - 失敗しても自分のPC内の作業が壊れるだけなので、気軽に何度も行えます。
    - イメージ：「ゲームのセーブポイントを作る」

  ### プッシュ（Push）：「インターネット上の場所」へ送信（アップロード）する操作
  - プッシュは、自分のPCで記録したコミットの内容を、インターネット上のリモートリポジトリ（GitHubなど）に送り出し、同期させる操作です。
    - 目的：自分の作業内容をサーバーに上げ、バックアップを取ったり、チームメンバーに公開したりする。
    - 特徴：
      - インターネット環境が必要です。
      - プッシュすると、他のメンバーがあなたの変更を見ることができるようになります。
    - イメージ：「作成したセーブデータを、みんながアクセスできるネット上のクラウドにアップロードする」

## コミットのメッセージはどのように書いてあげるのが最適でしょうか？
  - コミットメッセージは、後から自分やチームメンバーが見返した時に **「何をしたのか」「なぜしたのか」** がすぐに分かるように書くのが最適です。
  - コミットメッセージには「そのコミットが何に関する変更なのか」を短く示すためのタグで **プレフィックス(Prefix)** というものがあります。
    - プレフィックス(Prefix)とは接頭辞で、コミットメッセージの先頭に特定の文字をつけることです。  
    ```uml
    【例】
    feat : 〇〇の機能を追加
    refactor : 〇〇の機能をリファクタ
    fix : 〇〇で起きていたバグを修正
    ```
    -  いくつかプレフィックスに種類がありますので覚えておきましょう。
    
  |プレフィックス|用途・意味|
  | :---:  | :--- |
  | add | ファイル・コードの追加 |
  | change | ファイル・コードの変更|
  | remove| ファイルなどの削除 |
  | feat | 新機能の追加 |
  | docs | ドキュメントのみの変更（READMEなど） |
  | style | コードの動作に影響しない修正（空白、フォーマット、セミコロンの欠落など） |
  | refactor | バグ修正でも機能追加でもないコード変更（リファクタリング） |
  | perf | パフォーマンスを向上させるためのコード変更 |
  | test | テストの追加や修正 |
  | chore | ビルドプロセスや補助ツールの変更（設定ファイル変更など） |

## ローカルでマージするフローと、プルリクエストでマージするフローの違いは何でしょうか？
  ### ローカルでマージするフロー（個人開発・小規模向け）
  - 自分自身のPC（ローカルリポジトリ）でブランチを切り替え、コマンドを打って統合する方法です。
    - フロー：
      1. 作業用ブランチで作業し、コミットする。
      2. `main` ブランチに切り替える。
      3. `git merge` コマンドで作業用ブランチを統合する。
      4. プッシュしてリモートに反映させる。
    - メリット：手順が単純で、自分一人なら非常に高速です。
    - デメリット：「第三者によるレビュー」が発生しません。 **間違ったコードやバグを含んだままマージしてしまうリスクがあります。**

  ### プルリクエストでマージするフロー（チーム開発向け）
  - リモートリポジトリ（GitHubなど）の機能を利用して、「マージしてもいいか？」をチームメンバーに問う（リクエストする）方法です。
    - フロー:
      1. 作業用ブランチをリモートにプッシュする。
      2. GitHub上で「プルリクエスト」を作成する。
      3. チームメンバーがコードを読み、指摘（レビュー）を行う。
      4. 指摘があれば修正し、問題なければ「承認（Approve）」をもらう。
      5. GitHub上のボタンを押してマージする。
    - メリット:
      - コード品質の担保: 誰かの目でチェックが入るため、ミスを防げる。
      - ナレッジ共有: メンバー間で「どんなコードを書いたか」を知ることができる。
      - 履歴管理: 「なぜその変更が行われたか」という議論の記録がPRに残る。
    - デメリット: レビューを待つ時間が発生するため、**一人で行うよりは時間がかかる。**

## コンフリクトを起こしてしまった場合、どう対処すべきですか？
コンフリクトが発生した時、以下の3通りの方法で変更を取り込む必要があります。
### 1. 先にマージされた変更内容を取り込む
### 2. 後にマージしようとしている変更内容を取り込む
### 3. どちらの変更内容も取り込む
どれを選択するかに関しては状況によって異なりますが、特に3の両方取り込む場合は注意が必要です。
コンフリクトのメッセージは**文字列としてソースコードに書き込まれています。**
なのでエディタを使って両方の変更内容を取り込んだ処理で上書きすることで両方を取り込むことができます。

取り込み作業が終わったらコンフリクト解決で行った作業内容をコミットしましょう。
